## 项目概述

**ChengbookTTS v2.0.0** — 统一多模型 TTS 平台，集成 CosyVoice3、SoulX-Podcast、IndexTTS 三大模型，通过统一 API 提供语音合成服务。主要服务于"诚太乙"AI 教练系统。

## 架构

```
chengbook_tts/
├── engines/          # 引擎层 (base → cosyvoice3/soulxpodcast/indextts_v1/indextts_v2)
├── server/           # FastAPI 服务 (routes/, console.py, concurrency.py, history.py)
├── cli/              # 命令行工具
├── client/           # Python SDK
├── config/           # 配置 (settings.py, voices.py, emotions.py, model_configs/*.yaml)
├── vendor/           # 外部模型源码 (cosyvoice, matcha, soulxpodcast, indextts)
├── prompts/          # 音色提示音频
└── profiles/         # 用户偏好持久化
```

### 引擎能力矩阵

| 引擎 | 采样率 | 流式 | 情绪 | 语速控制 | 分词 |
|------|--------|------|------|----------|------|
| CosyVoice3 | 24000 | ✅ | ✅ | ✅ | ✅ |
| SoulX-Podcast | 24000 | ❌ | ✅(采样参数) | ❌ | ❌ |
| IndexTTS v1 | 24000 | ❌ | ❌ | ❌ | ❌ |
| IndexTTS v2 | 22050 | ❌ | ✅(8维向量) | ❌ | ❌ |

### API 端点

- `/api/health` — 健康检查 + 能力声明
- `/api/voices` / `/api/emotions` — 音色/情绪列表
- `/api/tts` — 非流式合成 → WAV
- `/api/tts/stream` — 流式合成 → PCM raw
- `/api/voices/custom` — 自定义音色上传/删除
- `/api/models` / `/api/model/switch` — 模型管理
- `/api/profile` — 用户偏好持久化
- `/api/history` — 合成记录展板
- `/v1/audio/speech` — OpenAI 兼容端点
- `/` — Web 控制台 (内联 SPA)

### 关键设计决策

1. **单例 ModelManager** — 同一时刻只有一个模型在 GPU 上，切换时自动卸载/加载
2. **预缓存** — CosyVoice3 启动时预计算所有音色特征和情绪 token
3. **能力声明** — 每个引擎声明 capability，前端据此灰显 UI
4. **情绪抽象** — 8 种通用情绪，各模型用不同机制实现
5. **OpenAI 兼容** — `/v1/audio/speech` 供诚太乙等外部产品调用
6. **并发控制** — asyncio.Semaphore + ThreadPoolExecutor，默认串行(GPU安全)
7. **自定义音色持久化** — custom_voices.json + WAV 文件，重启自动恢复

## 2026-07-06 (第二次会话): WebUI 转录文本输入框条件禁用

### 问题
WebUI 自定义音色上传区的"转录文本"输入框在所有模型下都可用，但实际上只有 SoulXPodcast 引擎会使用 `prompt_text`。CosyVoice3 的 `register_voice()` 虽然接受 `prompt_text` 参数但完全忽略它（只提取音频特征），IndexTTS v1/v2 也不使用。

### 改动
**console.py** (Web 控制台内联 SPA):

1. **CSS** (`input:disabled`): 新增禁用输入框样式 `opacity: 0.35; cursor: not-allowed; pointer-events: none`，与现有 `section-disabled` 风格一致。

2. **HTML**: 转录文本输入框下方新增警告提示 `<span id="promptTextNote">`，默认隐藏，非 SoulXPodcast 时显示 "⚠ 当前模型不支持转录文本，仅 SoulXPodcast 可用"。

3. **`updateCapUI()`**: 新增转录文本输入框的状态管理：
   - 检查 `activeModel === 'soulxpodcast'`
   - 不支持时：禁用输入框 + 更换 placeholder 为 "当前模型不支持转录文本（仅 SoulXPodcast 可用）" + 显示红色警告
   - 支持时：恢复可用状态 + 原始 placeholder + 隐藏警告

4. **`uploadVoice()`**: 仅在 `activeModel === 'soulxpodcast'` 时才将 `prompt_text` 附加到 FormData，防止无效数据发送到后端。

### 设计决策
选择**灰掉（disabled）**而非隐藏，原因：
- 与现有的 `section-disabled` 模式一致（情绪/语速/分词在不支持时也是灰掉整个卡片）
- 用户能看到功能存在，只是当前模型不支持
- 切换模型后自动恢复，无需重新发现功能
- 布局保持稳定

## 2026-07-06 (第三次会话): SoulXPodcast 拟人化文本预处理

### 需求
为 SoulXPodcast 引擎添加拟人化预处理，注入副语言标签（`<|laughter|>` 等）和随机说话瑕疵，摒弃"完美无瑕"的语音，模拟真人不完美感。

### 分析：SoulX-Podcast 原生副语言标签
原 SoulX-Podcast 模型在训练时已学会 5 种副语言标签，直接嵌入文本即可产生对应音频效果：

| 标签 | 效果 | 权重 |
|------|------|------|
| `<|breathing|>` | 呼吸声 | 8 |
| `<|laughter|>` | 笑声 | 5 |
| `<|sigh|>` | 叹气 | 3 |
| `<|coughing|>` | 咳嗽 | 2 |
| `<|throat_clearing|>` | 清嗓子 | 2 |

### 新增文件

**`chengbook_tts/engines/soulxpodcast_humanize.py`** (~200行)
- `HumanizeConfig` — 配置数据类（enabled, tag_probability, imperfection_probability 等）
- `HUMANIZE_PRESETS` — 三种预设强度：light(8%/3%), moderate(15%/8%), heavy(25%/15%)
- `apply_humanization()` — 主入口，注入副语言标签 + 说话瑕疵
- `perturb_sampling_params()` — 对 SamplingParams 施加 ±jitter
- `build_config()` — 从 humanize(bool) + level(str) 构建配置

### 拟人化策略

**a) 副语言标签注入**：在标点符号后按概率随机插入标签，加权随机选择（breathing 最常见），仅在中文上下文附近注入。

**b) 说话瑕疵**（从句边界插入，不在词内修改）：
- 填充词 (65%): `嗯，` / `啊，` / `那个，` / `就是说，` / `然后呢，`
- 口吃/重复 (20%): `这个这个` / `就就` / `然后然后` / `那那` / `这这`
- 纠错标记 (15%): `嗯...不是，` / `不对，应该是，` / `呃等一下，` / `啊说错了，`

**c) SamplingParams 扰动**：在情绪映射后的参数上施加 ±jitter，模拟真人语速/语调波动。

### 修改文件

| 文件 | 改动 |
|------|------|
| `chengbook_tts/engines/soulxpodcast.py` | synthesize() 中注入 humanization 预处理 + `_build_humanize_config()` |
| `chengbook_tts/server/models.py` | TTSRequest + ProfileRequest 加 `humanize`, `humanize_level` |
| `chengbook_tts/server/routes/native.py` | `_run_synthesize` 用 `functools.partial` 传递 humanize kwargs |
| `chengbook_tts/server/routes/profile.py` | update_profile 合并 humanize + humanizeLevel |
| `chengbook_tts/server/console.py` | 新增拟人化预处理卡片（开关+三级强度），仅 SoulXPodcast 可见 |
| `chengbook_tts/config/settings.py` | `HUMANIZE_ENABLED`, `HUMANIZE_LEVEL` 全局默认值 |

## 2026-07-06 (第四次会话): Humanize 日志可见性 + 合成超时 + 体验修复

### 问题
1. **日志不可见**：`apply_humanization()` 只在 `text != original_text` 时才打印日志，短文本随机没抽中时完全看不到，无法判断 humanize 是否被调用
2. **模型辨识困惑**：profile 中残留 `model_type: cosyvoice3`，用户误以为当前模型是 CosyVoice3
3. **长文本卡死**：194 字 heavy humanize 后涌入 7 个副语言标签 + 2 个瑕疵，SoulXPodcast 1.7B 推理时间过长，WebUI 一直显示"生成中"无反馈
4. **queue 计数异常**：卡住的请求不释放信号量，后续请求排队永远等待

### 改动

**`soulxpodcast.py`** — 日志策略调整：
- 只要 `humanize_config.enabled` 就始终打印预处理前后文本
- 有变化标记 `★`，无变化标记 `→(无变化)`
- 同时走 `logging.info()` + `print(flush=True)`，确保在 uvicorn 输出中可见

**`native.py`** — 两处改动：
- TTS 日志新增 `humanize=True/False` + `humanize_level` 显示
- `_run_synthesize()` 加 `asyncio.wait_for(timeout=settings.TTS_TIMEOUT)` 超时保护（默认 300s）
- 超时后返回 504 + 中文提示"合成超时，请缩短文本重试"
- `HTTPException` 异常直接透传，不被外层 `except Exception` 包成 500

**`settings.py`** — 新增：
- `TTS_TIMEOUT: int = 300` — 单次合成超时秒数，环境变量 `TTS_TIMEOUT` 可覆盖

### 验证
- Humanize heavy 预处理成功输出 `[Humanize] ★` 日志，标签和瑕疵正确注入
- 短文本（~50字）合成可正常完成，长文本（~200字）仍在合理推理时间内但受超时保护
- 超时后 WebUI 不再无限等待，会收到明确错误提示

### 设计决策
- **超时而非截断**：不在 humanize 层面限制文本长度（用户可能有长文本需求），而是在合成层加超时兜底
- **504 而非 500**：超时是网关层预期内的情况，504 Gateway Timeout 语义更准确
- **profile 中 model_type 不清理**：它是持久化快照不影响运行模型，未来可用于"记住上次使用的模型"功能

## 2026-07-06 (第五次会话): 拟人化策略优化

### 问题
1. **`<|laughter|>` 标签不合适**：客户投诉/"生气"等非轻松场景下笑声有讽刺意味，应当移除
2. **双标点**：瑕疵模板自带尾部标点（`嗯，`），插入后与原文标点叠加产生 `嗯，，`
3. **句尾赘语**：`_apply_imperfections` 在最后一个从句后也插入瑕疵，产生 `换平台了那那。` 这种语病
4. **口吃在从句边界无效**：`那那`/`这个这个` 需要紧跟被重复的词才自然，跨从句插入破坏语义

### 改动

**`soulxpodcast_humanize.py`** — 策略优化：

| 改动 | 说明 |
|------|------|
| 移除 `<|laughter|>` | 标签表从 5 个减至 4 个，权重重新分配 |
| 瑕疵模板去标点 | `嗯，`→`嗯`，`那个，`→`那个`，标点由原文负责 |
| 移除口吃类型 | 删除 `_STUTTERS` 模板和 `stutter_rate` 配置项 |
| 精简纠错标记 | 5 个→3 个，保留简短自然的（`嗯...不是` 等） |
| 限制插入位置 | 新增 `_MID_SENTENCE_PUNCT`，仅 `，、；：` 前可插入，句末 `。！？` 不可 |
| 修复 clause 索引 | 利用 `re.split` 捕获组特性：偶数索引=内容从句，奇数索引=分隔符，精确控制注入点 |
| 瑕疵比例调整 | `filled_pause_rate` 65%→85%，`self_correction_rate` 15% 不变 |

**验证**：3 次随机测试通过，无 `<|laughter|>`、无双标点、无句尾赘语。

## 2026-07-06 (第六次会话): 合成卡死 + 显存泄漏修复

### 问题
1. **合成卡死**：SoulXPodcast 1.7B `forward_longform` 对长文本推理卡死，进程无响应
2. **显存不释放**：Ctrl+C 终止后 GPU 显存仍被占用（WSL2 + CUDA）
3. **超时后线程残留**：`asyncio.wait_for` 取消的只是 asyncio 等待，底层线程仍在 GPU 上跑
4. **并发踩踏**：超时后信号量释放，下一个请求起新线程，两个线程同时抢 GPU
5. **无自愈能力**：一旦卡死只能重启，无自动恢复机制

### 根因分析

```
API 请求 → asyncio.wait_for(300s) → loop.run_in_executor → Thread → forward_longform
                                    ↑ 超时后 asyncio 取消等待     ↑ 线程继续占用 GPU
                                    HTTP 504 返回给客户端          ↑ Python 无法强杀线程
                                                                    ↑ 信号量已释放，下个请求可再起线程 → GPU OOM
```

### 修复: 五层防御体系

#### 第一层: 文本分段（预防）
**`soulxpodcast.py`** — 在 `synthesize()` 中将长文本按句末标点拆成 ≤120 字的小段：
- `_split_text_chunks()` 按 `。！？\n` 切分，保留标点在句尾
- 超长单句按 `，、；：` 二次拆分
- 每段独立调用 `_synthesize_chunk()` 推理
- 最终拼接时加 0.15s 静音间隔

#### 第二层: GPU 独占锁（隔离）
**`base.py`** — `TTSEngine.__init__` 新增：
- `_busy_lock = threading.Lock()` — 同一时刻只允许一个线程使用 GPU
- `acquire_gpu(timeout=5.0)` — 获取锁，失败则拒绝新请求（不踩踏）
- `release_gpu()` — 推理完成后释放

#### 第三层: 脏标记 + 自愈（恢复）
**`base.py`** — 新增 dirty/recovery API：
- `mark_dirty()` — 超时后标记引擎为脏
- `is_dirty` 属性
- `ensure_ready()` — 下次请求自动 unload + reinitialize
**`soulxpodcast.py`** — `synthesize()` 中调用 `ensure_ready()` + `torch.cuda.synchronize()` 提前捕获残留错误

#### 第四层: 超时清理（止损）
**`native.py`** — `_run_synthesize()` 超时处理增强：
- 标记 `engine.mark_dirty()`
- 调用 `torch.cuda.empty_cache()` 清理 GPU 碎片
- 返回 504 提示"引擎已自动重置，请重试"

#### 第五层: 进程退出清理（兜底）
**`app.py`** — 三层退出保障：
- `atexit.register(_emergency_gpu_purge)` — 任何退出路径都会清理 GPU
- `signal.signal(SIGINT/SIGTERM, _signal_handler)` — 覆盖 uvicorn 的 handler，先清 GPU 再 `sys.exit(0)`
**`concurrency.py`** — `shutdown(wait=False)` 不等待卡住的线程
**`cli/commands/serve.py`** — 启动前 `torch.cuda.empty_cache()` + `synchronize()` 清理上次残留

### 完整请求生命周期

```
请求进入
  → asyncio.Semaphore（限流）
  → ThreadPoolExecutor
    → engine.acquire_gpu(timeout=5s)    ← 防御层2: GPU 互斥
      → engine.ensure_ready()           ← 防御层3: 自愈检查
        → torch.cuda.synchronize()      ← 防御层3: 提前发现错误
        → _split_text_chunks(text)      ← 防御层1: 文本分段
        → for chunk in chunks:
            _synthesize_chunk(chunk)     ← 每段 ≤120 字，不卡死
      → engine.release_gpu()
      → torch.cuda.empty_cache()
    → 拼接音频 → 返回
超时路径:
  → engine.mark_dirty()                 ← 防御层4: 标记脏
  → torch.cuda.empty_cache()
  → HTTP 504 "引擎已自动重置，请重试"
退出路径:
  → atexit → GPU purge                 ← 防御层5: 兜底
  → SIGINT/SIGTERM → GPU purge → exit
```

| 文件 | 改动 |
|------|------|

| `chengbook_tts/engines/soulxpodcast.py` | 新增 `_MAX_CHUNK_CHARS=120`, `_split_text_chunks()`, `_split_long_sentence()`, `_synthesize_chunk()`; `synthesize()` 重构为 acquire_gpu + ensure_ready + 分段推理 + release_gpu |
